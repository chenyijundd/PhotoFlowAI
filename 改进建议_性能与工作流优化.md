## 🔴 P0 — 直接影响 80% 省时目标

### 1. 重复检测算法优化 ✅ 已完成

**现状**：[duplicate_detector/service.py](backend/ai/duplicate_detector/service.py) 对所有进入 Step 4 的照片做 dHash + 汉明距离两两比较 — O(n²)。级联管线已缩小候选集，但剩余仍需 ~880 万次比较（4200 张）。

**方案**（三步，逐步缩小）：

- **时间窗口预分组（最关键）**：按 EXIF 时间聚类。同一 10 秒窗口内的照片进入同一候选桶。4200 张 → ~50 桶 × ~84 张/桶 → 候选对从 880 万降到 ~17 万。
- **dHash 多索引哈希表**：每桶内仅对汉明距离 ≤ 10 的照片对做精确比较。每桶 84 张 → ~200-500 候选对。
- **SSIM 终判**：仅对通过前两步的候选对（预计 5000-15000 对）做精确判定。

**预计省时**：90-95% | **工作量**：3-4h（纯算法改动，不涉及 UI）

## 🟠 P1 — 摄影师工作流提速

### 2. 全尺寸图动态缩放 ✅ 已完成

**现状**：[detail_service.py](backend/api/detail_service.py) `/api/fullsize/{image_id}` 完整原图通过 `FileResponse` 传输。24MP JPEG ≈ 10MB，右侧详情面板实际只需约 800px 宽。

**问题**：传输完整原图再前端缩放，浪费带宽（10MB vs 200KB）和加载时间。

**方案**：
- 支持 `?width=` 参数：`/api/fullsize/{id}?width=800` 返回 Pillow 实时缩放后的 JPEG
- 或在导入时预生成"审片级大图"（长边 2560px）
- 仅在导出时使用完整原图

**预计省时**：详情面板加载速度 10-50× 提升 | **工作量**：1-2h

---

### 3. 筛选器 AI 计数合并 ✅ 已完成

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) AI 类别计数（模糊/重复/连拍/最佳/闭眼）通过 5 个独立 HTTP 请求获取。基本计数器已通过 `/api/photos/counts` 单次获取。

**方案**：将 5 个独立的 `/api/photos/{type}/count` 合并为 `/api/photos/counts` 的额外字段，单次 DB 扫描统计全部 9 个计数。

**预计省时**：5 次 HTTP → 1 次 | **工作量**：30min

## 🟡 P2 — 性能基础设施

### 4. 数据库连接池 + 连接复用 ✅ 已完成

**现状**：[connection.py](database/connection.py) `DatabaseConnection` 是 context manager，每次 `with DatabaseConnection()` 创建新连接。SQLite WAL 模式已启用，但没有连接复用。

**方案**：按线程缓存连接或实现简易连接池（单写 + N 读），复用而非每次创建。

**工作量**：1-2h

---

### 5. patch_scores 缓存

**现状**：[blur_detector_v2/detector.py](backend/ai/blur_detector_v2/detector.py) 16 区域拉普拉斯方差每次重新计算。如果调整阈值重分析，全部重算——但 patch_scores 是中间结果，与阈值无关。

**方案**：将 patch_scores 缓存到数据库 JSON 字段。首次计算 + 判定；后续仅用新阈值重新判定（毫秒级）。照片 hash 变更时失效。

**工作量**：1-2h

---

### 6. React 渲染优化

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) ~26 个独立 `useState` + 大量回调，状态更新可能触发不必要的子树重渲染。

**方案**：`useReducer` 替代多个独立 `useState`、拆分 Context、评估虚拟滚动方案。

**工作量**：2-3h
