# PhotoFlow AI — 改进建议：性能与工作流优化

> 基于产品作战文档 + 代码库全面审阅 | 2026-05-30

---

## 核心结论

当前最关键的瓶颈在 **AI 处理吞吐量**（P0 三项）。优化后预计将 5000 张照片的完整分析从 **20-30 分钟降到 5-8 分钟**，真正接近"10 分钟完成初筛"的产品目标。

P1 的流式输出 + 预加载则消除等待感，让摄影师**感到**比实际更快。

---

## 🔴 P0 — 直接影响 80% 省时目标

### 1. AI 分析并行化

**现状**：[ai_service.py](backend/api/ai_service.py) 四个检测步骤串行执行，模糊检测逐张处理 5000 张照片。每张照片独立计算拉普拉斯方差，天然可并行，但当前仅用单线程。

**问题**：4 核 CPU 只用了 1 核，白白浪费算力。

**方案**：
- 模糊检测改用 `concurrent.futures.ProcessPoolExecutor`，将 `photo_ids` 分块给多个进程并行计算
- 每个进程独立打开 OpenCV 读取图像、计算拉普拉斯、返回结果
- 主线程负责汇总结果并写入 SQLite（写操作保持单线程避免锁冲突）
- N 核 CPU → 接近 N 倍提速

**预计省时**：50-70%

---

### 2. 重复检测算法优化

**现状**：对所有照片做 dHash + SSIM 两两比较 — O(n²)。5000 张 = 1250 万次比较，不可行。

**问题**：没有利用照片的时间属性和感知哈希的索引特性来缩小候选集。

**方案**：
- **第一步**：EXIF 时间窗口预分组（同一秒或相邻秒拍摄的照片才进候选集）— 连拍才可能重复
- **第二步**：dHash 汉明距离快速筛选。为所有照片的 dHash 建立多索引哈希表（multi-index hashing），仅对汉明距离 ≤ 10 的照片对做 SSIM
- **第三步**：仅对候选对做 SSIM 精确判定

**预计省时**：60-80%（候选集从 1250万 降到 ~5000-10000）

---

### 3. 缩略图级 AI 预处理

**现状**：[blur_detector_v2/detector.py](backend/ai/blur_detector_v2/detector.py) 每次对完整原图（2400万+ 像素）做 4×4 拉普拉斯。

**问题**：拉普拉斯对分辨率不敏感（它检测的是边缘梯度），缩略图（长边 800px）的结果与原图高度相关，但处理速度快 20-50 倍。

**方案**：
- 先用缩略图（长边 800px）快速预筛，分三档：
  - 得分 > 80 → **直接通过**（明显清晰）
  - 得分 < 25 → **直接标记模糊**（明显模糊）
  - 25 ≤ 得分 ≤ 80 → **边界案例**，回退到原图精度分析
- 缩略图已在缓存中，无额外 I/O 开销

**预计省时**：40-60%（大部分照片落在大分/小分段，无需触碰原图）

## 🟠 P1 — 摄影师工作流提速

### 4. 边分析边审片（流式结果输出）

**现状**：必须等待全部 4 步 AI 分析完成后才能开始审片。5000 张照片分析可能需要 20-30 分钟，摄影师干等。

**问题**：模糊检测和重复检测是独立步骤，不需要互相等待即可产生可审阅的结果。

**方案**：
- 模糊检测完成一张立即更新 `is_blur` 字段 + 通过 WebSocket 推送更新
- 摄影师可以在模糊检测进行中就开始浏览"废片"筛选器
- 后续步骤（重复检测、连拍分组）的分析结果持续追加到 UI
- 进度条改为"阶段进度指示器"而非阻塞式等待

**预计省时**：消除 20-30 分钟等待焦虑，实际审片与 AI 分析并行

---

### 5. 智能预加载（零延迟审片体验）

**现状**：[DetailPanel.tsx](frontend/src/components/DetailPanel.tsx) 每次切换照片都重新请求 API + 从硬盘加载完整原图，有明显的加载闪烁。

**问题**：摄影师快速浏览时（1-2 秒/张），加载延迟打断节奏。

**方案**：
- 预加载当前可见区域 ±2 行（约 20 张）的全尺寸图到内存
- 缩略图全部预加载到浏览器 IndexedDB，初次加载后永远从本地读取
- 键盘导航时异步预取下一张，确保切换时已有缓存
- 实现 LRU 缓存淘汰（内存上限 200MB）

**预计省时**：审片延迟降到 0，节奏不被打断

---

### 6. 连拍组多图同屏对比

**现状**：ComparePage 仅支持 2 张 A/B 对比，连拍组可能有 5-15 张。需要反复切换比较。

**问题**：摄影师需要同时看 3-5 张连拍的表情/姿态差异，逐一对比效率极低。

**方案**：
- 新增"连拍组网格对比"模式：选中连拍组 → 显示 N 张照片的网格布局（自动适配 2×2、2×3、3×3）
- 所有照片同步缩放和平移（鼠标悬停在哪张，全部跟随）
- 点击一张标记为"+1"最佳，支持多张选中后导出
- 从连拍胶卷（BurstFilmstrip）一键进入

**预计省时**：连拍组选片效率 +50%

---

### 7. 筛选器计数缓存

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) 每次切换筛选器都发独立 API 请求重新 count。

**问题**：4 个筛选器标签的计数各自发起 HTTP 请求，切换有可感知延迟。

**方案**：
- 后端返回一次全量计数（单次 SQL 查询）：`{ all, starred, unprocessed, rejected }`
- 前端缓存，仅在 star/reject 操作后递增更新（乐观更新）
- 引入 stale-while-revalidate：先用缓存值，后台异步校准

**预计省时**：筛选器切换零延迟

---

### 8. 批量操作撤销 (Undo)

**现状**：一键选片 `acceptAllBest` / `rejectAllRest` 是破坏性操作，没有撤销。操作覆盖 100+ 张照片，一失手就要逐张恢复。

**问题**：破坏性操作无保护，摄影师心理负担重，不敢用"一键选片"。

**方案**：
- 实现操作历史栈（最近 50 条操作记录）
- 每条记录存储：操作类型、涉及的 `image_id` 列表、操作前后的 `star_rating` / `is_rejected` 值
- 支持 Ctrl+Z 单步撤销、Ctrl+Shift+Z 重做
- 批量操作在数据库层面用事务包裹，撤销 = 反向 UPDATE

**预计省时**：消除"一键选片"的心理障碍，防误操作焦虑

## 🟡 P2 — 性能基础设施

### 9. 数据库连接池 + WAL 模式

**现状**：[repository.py](database/repository.py) 每个请求 `PhotoRepository()` → `DatabaseConnection()` → 新建 SQLite 连接。高并发下锁竞争严重（多个请求同时尝试写）。

**方案**：
- 开启 SQLite WAL 模式（Write-Ahead Logging），支持一写多读并发
- 实现简易连接池（单写连接 + 3 读连接），复用连接而非每次创建
- 或者评估迁移到 DuckDB（列式存储，分析查询更快）

---

### 10. WebSocket 替代轮询

**现状**：[ai_service.py](backend/api/ai_service.py) 后台线程 + 前端 `setInterval` 轮询进度。轮询间隔 = 延迟，且大部分轮询返回相同数据。

**方案**：
- FastAPI WebSocket 端点：`/ws/ai/progress/{task_id}`
- 后台线程每完成一张照片，通过 asyncio 事件推送到 WebSocket
- 前端用事件驱动更新进度条，消除轮询开销
- 降级方案：WebSocket 不可用时回退到轮询

---

### 11. 全尺寸图动态缩放

**现状**：[detail_service.py:129-146](backend/api/detail_service.py#L129-L146) 完整原图通过 `FileResponse` 传输。24MP JPEG ≈ 10MB，右侧详情面板实际只需 800px 宽。

**问题**：传输完整原图到前端再缩放显示，浪费带宽和加载时间。

**方案**：
- 支持 `?width=` 参数：`/api/fullsize/{id}?width=800` 返回 Pillow 实时缩放后的 JPEG
- 或在导入时预生成"预览级大图"（长边 2560px），作为审片专用分辨率
- 仅在导出时使用完整原图

---

### 12. patch_scores 缓存

**现状**：每次分析都对同一张照片重新计算拉普拉斯。如果摄影师调整阈值重新分析，全部重算。

**问题**：`patch_scores`（16 个区域的拉普拉斯方差）是纯中间结果，与阈值无关，但每次都重新计算。

**方案**：
- 将 `patch_scores` 缓存到数据库（JSON 字段或独立表）
- 首次分析 → 计算 patch_scores + 用当前阈值判定
- 后续调整阈值 → 只重新判定（纯数值比较，毫秒级），无需重新计算
- 仅在照片文件变更（hash 不同）时失效缓存

---

### 13. React 渲染优化

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) 约 20 个 useState + 大量回调。状态更新可能触发不必要的子树重渲染。

**方案**：
- 拆分 Context：数据 Context（photos/total/loading）与 UI Context（filter/selection）分离
- 用 `useReducer` 替代多个独立 `useState`，批量 dispatch
- 将 `loadStarredCount` / `loadRejectedCount` 等计数查询合并为单次请求
- 评估 `@tanstack/react-virtual` 替代 `react-window`（更好的 TypeScript 支持和更小的包体积）

## 🟢 P3 — 摄影专用体验增强

### 14. 时间线视图

摄影师的思维模式是"拍摄时间线"。增加按 EXIF 时间排序的时间轴浏览模式，替代纯网格。左侧时间轴 + 右侧照片流，支持按小时/场景跳转。

---

### 15. 选片进度仪表盘

实时显示统计面板：
```
已选 52/5000 (1%)  |  废片 205 (4%)  |  待处理 4743 (95%)
预计还需 25 分钟  |  当前速度 3.2 张/秒
```
给摄影师明确的进度预期，减少焦虑。

---

### 16. 导出预设

不同客户（婚礼/儿童/商业）有不同导出规格（分辨率/格式/命名规则/水印）。支持保存和加载导出配置模板，一键切换。

---

### 17. 键盘长按批量标记

摄影师审片节奏很快。Space（加星）/ X（废片）长按时自动连续标记（类似游戏按键），而不是逐张点击。长按 300ms 后触发连续模式，每 100ms 标记一张。

---

### 18. RAW 文件嵌入式预览

文档提到 RAW 仅支持预览。大多数 RAW 格式（CR2/CR3/NEF/ARW）内含全尺寸 JPEG 预览。直接提取嵌入式 JPEG 而非解 RAW，速度提升 50-100 倍。

---

### 19. 颜色标签系统

扩展当前"星标/废片"二元分类为中粒度分级：

| 颜色 | 含义 | 用途 |
|------|------|------|
| 🔴 红 | 必选 | 客户指定 / 关键瞬间 |
| 🟡 黄 | 备选 | 可修可用 |
| 🟢 绿 | 精修 | 需要深度修图 |
| 🔵 蓝 | 排版 | 适合相册排版 |
| 🟣 紫 | 待定 | 犹豫不决，稍后决定 |

快捷键：数字键 1-5 设置对应颜色标签。

---

### 20. 多项目管理 / 工作区

当前为单项目设计。工作室同时处理 3-5 场婚礼。增加项目切换功能，每个项目使用独立 SQLite 数据库文件，支持归档和重新打开。

---

## 🔵 P4 — 健壮性与协作

### 21. 断点续扫

AI 分析中途崩溃后，重新分析能从上次中断处继续，而非重来。实现方式：在数据库中记录每张照片的 `blur_checked_at` / `duplicate_checked_at` 时间戳，重新分析时跳过已完成的照片。

---

### 22. 自动备份

定期自动备份 SQLite 数据库（保留最近 5 个版本），防止选片结果意外丢失。每小时自动备份一次，存储在 `backups/` 目录，文件名带时间戳。支持手动触发备份和从备份恢复。

---

### 23. 选片报告导出

导出精选照片时附带一份 HTML/PDF 选片报告，包含：
- 选片统计（总数/选中/废片/模糊数）
- 精选照片缩略图网格
- 废片原因标注（模糊/闭眼/重复/手动标记）
- 连拍组对比摘要

供客户确认选片结果，减少沟通成本。
