# PhotoFlow AI — 改进建议：性能与工作流优化

> 基于产品作战文档 + 代码库全面审阅 | 2026-06-02（已更新）

---

## 核心结论

当前版本（`328a0be`）已完成 10 项改进，P0 中模糊检测并行化、缩略图级预筛已落地，P1 流式输出已实现。剩余的关键瓶颈在 **重复检测算法**（P0，已部分实现）。优化后预计将 5000 张照片的完整分析从 **15-20 分钟降到 4-6 分钟**。

## ✅ 已完成（本次迭代已实现）

### 1. ~~AI 分析并行化~~ ✅ DONE

**现状**：`backend/ai/blur_detector_v2/service.py` 已改用 `ProcessPoolExecutor`。`_process_blur_chunk()` 将 photo_ids 分块分发给多个进程并行计算拉普拉斯方差，主线程汇总结果并写入 SQLite。

- Worker 数量：`max(2, min(cpu_count, 8))`
- Chunk 策略：每个 worker ~2 个 chunk，保持所有核心持续忙碌
- DB 写操作：仅主线程执行，避免 SQLite 锁冲突
- 取消机制：`executor.shutdown(cancel_futures=True)` 即时终止

**效果**：4 核 → 模糊检测省时 60-70%

### 2. ~~批量操作撤销 (Undo/Redo)~~ ✅ DONE

**现状**：`frontend/src/context/UndoRedoContext.tsx` 完整实现。50 步操作历史栈，支持单步/批量撤销重做。`Ctrl+Z` 撤销、`Ctrl+Shift+Z` 重做。已在 `BrowserPage.tsx` 的加星/废片操作中集成。

### 3. ~~RAW 预览~~ ✅ DONE

**现状**：`backend/raw_preview/extractor.py` 支持 .CR2/.CR3/.NEF/.ARW/.DNG/.ORF/.RAF/.RW2/.PEF/.SRW/.RAW 共 11 种 RAW 格式。提取内嵌 JPEG 预览或转换 bitmap，缓存到 `cache/raw_previews/`。`read_image_bgr()` 统一入口，支持 OpenCV → PIL 回退。

### 4. ~~AI 分析统计摘要~~ ✅ DONE

**现状**：`frontend/src/components/AnalysisSummaryModal.tsx` 分析完成后弹出摘要。后端 `/api/ai/summary` 返回完整统计数据（闭眼/模糊/连拍/重复/最佳/正常）。

### 8. ~~连拍胶片条~~ ✅ DONE

**现状**：`frontend/src/components/BurstFilmstrip.tsx` 水平滚动缩略图条，支持键盘导航、保留最佳/全部保留/全部拒绝操作。

### 9. ~~缩略图级 AI 预处理~~ ✅ DONE

**现状**：[detector.py](backend/ai/blur_detector_v2/detector.py) 的 `quick_blur_check()` + [service.py](backend/ai/blur_detector_v2/service.py) 的 `_process_blur_chunk()` 实现了完整的三级流水线：

1. **Tier 1 — 800px AI 预览**：[preview_generator.py](backend/ai/ai_preview/preview_generator.py) 生成/缓存长边 800px 的 JPEG 预览，读取速度快 50-100×
2. **Tier 2 — 快速拉普拉斯预筛**：单次全局拉普拉斯方差，分三档裁决：
   - 得分 > 80 → **直接通过**（明显清晰，跳过原图分析）
   - 得分 < 25 → **直接标记模糊**（明显模糊，跳过原图分析）
   - 25 ≤ 得分 ≤ 80 → **边界案例**，回退到原图精度分析
3. **Tier 3 — 完整多区域分析**：仅边界案例触发 `calculate_blur_v2()` 的 4×4 多区域 + 中心加权 + top-50% 中值

**效果**：典型照片集约 80% 在 Tier 2 即完成分类，完全跳过昂贵的原图 I/O 和多区域计算。只有 ~20% 的边界案例需要完整分析。

## 🔴 P0 — 直接影响 80% 省时目标

### 1. 重复检测算法优化

**现状**：[duplicate_detector/service.py](backend/ai/duplicate_detector/service.py) 对所有进入 Step 4 的照片做 dHash + 汉明距离两两比较 — 严格的 O(n²)。

好消息是级联管线已大幅缩小候选集：

```
Step 1 闭眼检测 ──→ 标记废片，跳过后续所有步骤
Step 2 模糊检测 ──→ 标记废片，跳过后续所有步骤
Step 3 连拍分组 ──→ 组内选最佳，整组跳过重复检测
Step 4 重复检测 ←── 仅处理"干净"照片（无闭眼、无模糊、非连拍）
```

级联后候选集变化（5000 张场景）：

| 阶段 | 剩余照片 | 排除原因 |
|------|---------|---------|
| 原始 | 5000 | — |
| 闭眼排除 | ~4950 | 闭眼 = 废片，不进入后续 |
| 模糊排除 | ~4800 | 模糊 = 废片，不进入后续 |
| 连拍排除 | ~4200 | 连拍组内已选最佳，组内其他照片不参与重复检测 |
| **进入重复检测** | **~4200** | 仍需 4200² / 2 ≈ **880 万次比较** |

**问题**：880 万次比较仍然太多，且有两个可以进一步利用的线索：

1. **EXIF 时间天然分段**：非连拍的照片如果拍摄间隔 > 30 秒，几乎不可能是重复照片（摄影师在拍不同场景）。利用时间窗口可以将 4200 张拆成数十个独立的小候选群。
2. **连拍组的"最佳"照片之间可能重复**：同一连拍组选出 1 张最佳，但不同连拍组的最佳照片如果是同一场景的不同角度，仍应被检测为重复。

**方案**（分三步，逐步缩小）：

- **第一步 — 时间窗口预分组（最关键）**：按 EXIF 时间将照片聚类。同一时间窗口（如 10 秒内）的照片进入同一候选桶。不同桶之间不做比较。4200 张 → 拆成 ~50 个桶，平均每桶 ~84 张 → 候选对从 880 万降到 ~17 万（50 × 84²/2）。

- **第二步 — dHash 多索引哈希表**：在每个时间桶内，为所有照片的 dHash 建立多索引哈希表（multi-index hashing）。仅对汉明距离 ≤ 10 的照片对做精确比较。每桶 84 张 → 候选对进一步降到 ~200-500。

- **第三步 — SSIM 终判**：仅对通过前两步的候选对（预计 5000-15000 对）做 SSIM 精确判定。

**与连拍分组的关系**：

```
连拍分组（Step 3）:           重复检测（Step 4）:
  ┌─────────────────┐           ┌─────────────────┐
  │ 同一场景、连拍快门  │           │ 不同时刻、同一场景  │
  │ 间隔 < 2 秒       │           │ 间隔 > 2 秒       │
  │ 选出 1 张最佳     │           │ 选出 1 张保留     │
  │ 如：10 连拍 → 1 张 │           │ 如：拍了 3 次 → 1 张 │
  └─────────────────┘           └─────────────────┘
      互补关系：连拍处理"太快"的重复，重复检测处理"太像"的冗余
```

**预计省时**：90-95%（候选对从 880 万降到 1-1.5 万，且 dHash 比 SSIM 快 100×）

**工作量**：3-4h。时间窗口和 dHash 索引是纯算法改动，不涉及 UI。

## 🟠 P1 — 摄影师工作流提速

### 3. 边分析边审片（流式结果输出）

**现状**：[ai_service.py](backend/api/ai_service.py) + [photo_service.py](backend/api/photo_service.py) 新增 SSE（Server-Sent Events）端点替代轮询，前端 [BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) 用 `EventSource` 替代 `setInterval`。

- **后端 SSE 端点**：
  - `GET /api/ai/analyze-stream/{task_id}` — 分析进度流，推送 `step_start` / `progress` / `step_complete` / `task_complete` / `task_cancelled` 事件
  - `GET /api/photos/cull-stream/{task_id}` — 选片进度流，同样事件模式
  - 通过 `queue.Queue`（线程安全）+ `asyncio.run_in_executor()` 实现跨线程事件传递
- **取消支持**：新增 `POST /api/ai/analyze-cancel/{task_id}` 和 `POST /api/photos/cull-cancel/{task_id}` 端点
- **前端增量 UI**：
  - AI 分类栏（闭眼/模糊/连拍/重复/最佳）在首个步骤完成后即出现，后续步骤完成时数字递增
  - 进度条显示取消按钮，分析过程中可随时中止
  - 分析进行中不阻塞照片浏览，可切换筛选器
  - 新出现的分类 chip 带脉冲动画

**效果**：消除 15-20 分钟等待焦虑，摄影师可在分析进行中开始审片。SSE 单连接替代每 400ms 轮询，减少 HTTP 开销。

---

### 4. 智能预加载（零延迟审片体验）

**现状**：[DetailPanel.tsx](frontend/src/components/DetailPanel.tsx) 每次切换照片都重新请求 API + 从硬盘加载完整原图，有明显的加载闪烁。ComparePage 有基本的 `new Image()` 预加载，但通用审片流程没有。

**问题**：摄影师快速浏览时（1-2 秒/张），加载延迟打断节奏。

**方案**：
- 预加载当前可见区域 ±2 行（约 20 张）的全尺寸图到内存
- 缩略图全部预加载到浏览器 IndexedDB，初次加载后永远从本地读取
- 键盘导航时异步预取下一张，确保切换时已有缓存
- 实现 LRU 缓存淘汰（内存上限 200MB）

**预计省时**：审片延迟降到 0，节奏不被打断

---

### 5. 连拍组多图同屏对比

**现状**：`BurstFilmstrip.tsx` 仅支持逐张浏览。`ComparePage` 仅支持 2 张 A/B 对比。连拍组可能有 5-15 张，需要反复切换比较。

**问题**：摄影师需要同时看 3-5 张连拍的表情/姿态差异，逐一对比效率极低。

**方案**：
- 新增"连拍组网格对比"模式：选中连拍组 → 显示 N 张照片的网格布局（自动适配 2×2、2×3、3×3）
- 所有照片同步缩放和平移（鼠标悬停在哪张，全部跟随）
- 点击一张标记为"+1"最佳，支持多张选中后导出
- 从连拍胶卷（BurstFilmstrip）一键进入

**预计省时**：连拍组选片效率 +50%

---

### 6. 筛选器 AI 计数合并

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) 基本筛选器（全部/已选/待处理/废片）已通过单次 `/api/photos/counts` 批量获取。但 AI 类别计数（模糊/重复/连拍/最佳/闭眼）仍通过 5 个独立 HTTP 请求获取。

**问题**：5 个独立请求 = 5 次数据库往返 + 网络开销，切换有可感知延迟。

**方案**：
- 新增后端端点 `/api/ai/counts`：单次 SQL 查询返回 `{ blur, duplicate, burst, best, closed_eye }`
- 前端一次请求获取全部 AI 计数，切换零延迟
- 与基本筛选器计数合并为一个端点更好：`/api/photos/all-counts`

**预计省时**：AI 筛选器切换延迟从 5 次往返降为 1 次

## 🟡 P2 — 性能基础设施

### 7. 数据库连接池 + 连接复用

**现状**：[connection.py](database/connection.py) 每次 `PhotoRepository()` 操作打开新连接。SQLite WAL 模式已启用（支持一写多读），但没有连接复用。

**方案**：
- 实现简易连接池（单写连接 + 3 读连接），复用连接而非每次创建
- 或在 `PhotoRepository` 中按线程缓存连接（thread-local storage）
- 评估迁移到 DuckDB（列式存储，分析查询更快，但引入新依赖）

---

### 8. WebSocket 替代轮询

**现状**：[ai_service.py](backend/api/ai_service.py) 后台线程 + 前端 `setInterval` 轮询进度。轮询间隔 = 延迟，且大部分轮询返回相同数据。

**方案**：
- FastAPI WebSocket 端点：`/ws/ai/progress/{task_id}`
- 后台线程每完成一批照片，通过 asyncio 事件推送到 WebSocket
- 前端用事件驱动更新进度条，消除轮询开销
- 降级方案：WebSocket 不可用时回退到轮询

---

### 9. 全尺寸图动态缩放

**现状**：[detail_service.py](backend/api/detail_service.py) `/api/fullsize/{image_id}` 完整原图通过 `FileResponse` 传输。24MP JPEG ≈ 10MB，右侧详情面板实际只需 800px 宽。

**问题**：传输完整原图到前端再缩放显示，浪费带宽和加载时间。

**方案**：
- 支持 `?width=` 参数：`/api/fullsize/{id}?width=800` 返回 Pillow 实时缩放后的 JPEG
- 或在导入时预生成"预览级大图"（长边 2560px），作为审片专用分辨率
- 仅在导出时使用完整原图

---

### 10. patch_scores 缓存

**现状**：[blur_detector_v2/detector.py](backend/ai/blur_detector_v2/detector.py) 每次分析都对同一张照片重新计算拉普拉斯。如果摄影师调整阈值重新分析，全部重算。

**问题**：`patch_scores`（16 个区域的拉普拉斯方差）是纯中间结果，与阈值无关，但每次都重新计算。

**方案**：
- 将 `patch_scores` 缓存到数据库（JSON 字段或独立表）
- 首次分析 → 计算 patch_scores + 用当前阈值判定
- 后续调整阈值 → 只重新判定（纯数值比较，毫秒级），无需重新计算
- 仅在照片文件变更（hash 不同）时失效缓存

---

### 11. React 渲染优化

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) 约 26 个独立 `useState` + 大量回调。状态更新可能触发不必要的子树重渲染。

**方案**：
- 拆分 Context：数据 Context（photos/total/loading）与 UI Context（filter/selection）分离
- 用 `useReducer` 替代多个独立 `useState`，批量 dispatch
- 将 AI 类别计数合并为单次请求（见 #6）
- 评估 `@tanstack/react-virtual` 替代 `react-window`

## 🔵 P4 — 健壮性与协作

### 18. 断点续扫

AI 分析中途崩溃后，重新分析能从上次中断处继续，而非重来。实现方式：在数据库中记录每张照片的 `blur_checked_at` / `duplicate_checked_at` 时间戳（已有 `analyzed_at` 字段），重新分析时跳过已完成的照片。配合 patch_scores 缓存（#10），阈值调整时无需重算。

---

### 19. 自动备份

定期自动备份 SQLite 数据库（保留最近 5 个版本），防止选片结果意外丢失。每小时自动备份一次，存储在 `backups/` 目录，文件名带时间戳。支持手动触发备份和从备份恢复。

---

### 20. 选片报告导出

导出精选照片时附带一份 HTML/PDF 选片报告，包含：
- 选片统计（总数/选中/废片/模糊数）
- 精选照片缩略图网格
- 废片原因标注（模糊/闭眼/重复/手动标记）
- 连拍组对比摘要

供客户确认选片结果，减少沟通成本。
