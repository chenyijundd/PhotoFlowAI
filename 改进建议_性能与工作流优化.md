## 🔴 P0 — 直接影响 80% 省时目标

### 1. 重复检测算法优化 ✅ 已完成

**现状**：[duplicate_detector/service.py](backend/ai/duplicate_detector/service.py) 对所有进入 Step 4 的照片做 dHash + 汉明距离两两比较 — O(n²)。级联管线已缩小候选集，但剩余仍需 ~880 万次比较（4200 张）。

**方案**（三步，逐步缩小）：

- **时间窗口预分组（最关键）**：按 EXIF 时间聚类。同一 10 秒窗口内的照片进入同一候选桶。4200 张 → ~50 桶 × ~84 张/桶 → 候选对从 880 万降到 ~17 万。
- **dHash 多索引哈希表**：每桶内仅对汉明距离 ≤ 10 的照片对做精确比较。每桶 84 张 → ~200-500 候选对。
- **SSIM 终判**：仅对通过前两步的候选对（预计 5000-15000 对）做精确判定。

**预计省时**：90-95% | **工作量**：3-4h（纯算法改动，不涉及 UI）

## 🟠 P1 — 摄影师工作流提速

### 2. 全尺寸图动态缩放 ✅ 已完成

**现状**：[detail_service.py](backend/api/detail_service.py) `/api/fullsize/{image_id}` 完整原图通过 `FileResponse` 传输。24MP JPEG ≈ 10MB，右侧详情面板实际只需约 800px 宽。

**问题**：传输完整原图再前端缩放，浪费带宽（10MB vs 200KB）和加载时间。

**方案**：
- 支持 `?width=` 参数：`/api/fullsize/{id}?width=800` 返回 Pillow 实时缩放后的 JPEG
- 或在导入时预生成"审片级大图"（长边 2560px）
- 仅在导出时使用完整原图

**预计省时**：详情面板加载速度 10-50× 提升 | **工作量**：1-2h

---

### 3. 筛选器 AI 计数合并 ✅ 已完成

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) AI 类别计数（模糊/重复/连拍/最佳/闭眼）通过 5 个独立 HTTP 请求获取。基本计数器已通过 `/api/photos/counts` 单次获取。

**方案**：将 5 个独立的 `/api/photos/{type}/count` 合并为 `/api/photos/counts` 的额外字段，单次 DB 扫描统计全部 9 个计数。

**预计省时**：5 次 HTTP → 1 次 | **工作量**：30min

## 🟡 P2 — 性能基础设施

### 4. 数据库连接池 + 连接复用 ✅ 已完成

**现状**：[connection.py](database/connection.py) `DatabaseConnection` 是 context manager，每次 `with DatabaseConnection()` 创建新连接。SQLite WAL 模式已启用，但没有连接复用。

**方案**：按线程缓存连接或实现简易连接池（单写 + N 读），复用而非每次创建。

**工作量**：1-2h

---

### 5. patch_scores 缓存 ✅ 已完成

**现状**：[blur_detector_v2/detector.py](backend/ai/blur_detector_v2/detector.py) 16 区域拉普拉斯方差每次重新计算。如果调整阈值重分析，全部重算——但 patch_scores 是中间结果，与阈值无关。

**方案**：将 patch_scores 缓存到数据库 JSON 字段。首次计算 + 判定；后续仅用新阈值重新判定（毫秒级）。照片 hash 变更时失效。

**工作量**：1-2h

---

### 6. React 渲染优化 ✅ 已完成

**现状**：[BrowserPage.tsx](frontend/src/pages/BrowserPage.tsx) ~26 个独立 `useState` + 大量回调，AI 分析期间 SSE 进度更新导致 BrowserPage 频繁 re-render，ImageGrid 子树跟着做无意义虚拟 DOM 计算。

**方案**：`ImageGrid` 加 `React.memo`，props 不变时跳过整个子树。AI 分析 SSE 进度推送不再触发网格重渲染。ImageCard 已有 `React.memo` + 自定义 comparator，leaf 层级已优化。

**工作量**：0.5h（实际 5 行代码）

### 17. 照片删除功能 ✅ 已完成

PhotoFlow AI 完全没有用户可触发的删除功能。当前唯一会删数据库记录的场景是导入时检测到文件已不存在（手动在资源管理器删除 → 重新导入 → Step 5 自动清理 DB 记录）。没有 API 端点、没有前端按钮、没有快捷键。
￼
利（Pros）
1. 补齐工作流闭环
当前流程是「导入 → AI 分析 → 筛选（打星/拒标）→ 导出」，中间缺少「清理」环节。用户标了一堆 reject，这些废片仍占用磁盘空间。删除是挑选流程的自然终点。
2. 释放存储空间
RAW 文件单张 30–80MB 不等，连拍组动辄几十张。不能删除意味着废片永久占据硬盘，用户只能手动去资源管理器操作，体验割裂。
3. 行业标配
Lightroom、Capture One、Photo Mechanic、DigiKam 等所有照片管理工具都有删除功能。没有它会被视为功能缺失。
4. 现有基础设施已就绪
• repo.delete_photo() 已实现
• 导入流程已有 stale record 清理逻辑
• send2trash 库（Python）只需 pip install，一行代码即可实现回收站删除
￼
弊（Risks）
1. 数据丢失不可逆
即使在回收站，用户清空后也无法恢复。这是照片管理工具中最危险的操作。
2. 实体关系完整性被破坏
关系类型
删除一张的后果
RAW+JPEG 配对
删除 RAW → JPEG 变成孤儿，配对断裂；删除 JPEG → RAW 丢失快速预览
重复组
组内成员减少，is_best_in_duplicate 可能指向已删除记录
连拍组
burst_position 序列断裂，burst_total 数字失效
AI 分析结果
is_best_in_burst / is_best_in_duplicate 可能悬空
3. 误操作风险
当前 UI 已有点击选中 + 快捷键操作（Space=打星, D=拒标）。如果 Delete 键直接删除，误触风险极高。
4. 缓存残留
缩略图、RAW 预览图等缓存文件需要同步清理，否则会积累垃圾文件。
5. 与导入的交互
如果用户删除了照片后又重新导入同一目录 → 照片重新出现。这在"从 SD 卡导入后清理"场景下是正确的，但可能让用户困惑。
￼
结论：应该做，但分两层
不要直接做"永久删除"，而是做一个「回收站」机制。
第一层：软删除（低风险，立即实施）
复用现有的 is_rejected 标志 + 新增一个 deleted_at 时间戳字段：
• 用户在详情面板或右键菜单选择「移到回收站」
• 后端设置 deleted_at = now()，照片从主视图消失
• 新增一个「回收站」筛选 tab，显示已标记删除的照片
• 在回收站中可以「还原」（清除 deleted_at）或「彻底删除」
• 30 天后自动彻底删除（可选）
• 键盘快捷键：Ctrl+Delete（故意不用单键，防止误触）
第二层：彻底删除（需确认，在回收站中操作）
• 仅在回收站视图中可触发
• 二次确认对话框，显示文件名 + 缩略图预览
• RAW+JPEG 配对：删除任意一张时提示「同时删除配对的 RAW/JPG 文件」，默认勾选
  注：目前不解释RAW，窗口上应该只有RAW的JPEG
• 连拍组 / 重复组：不强制联动，但显示「此照片属于 X 组（共 N 张）」
• 删除操作：
1. 调用 send2trash 将文件移到回收站（Windows）/ .Trash（macOS）
2. 删除 DB 记录
3. 清理缩略图缓存 + RAW 预览缓存
4. 修复关联实体（重新计算 burst_total、清理配对 ID 等）
5. 批量操作不需逐张确认，但显示清单
6. 删除照片按拍摄时间 (EXIF)排序
不做的
• ❌ 直接 os.remove() 永久删除（不可恢复，风险太大）
• ❌ 批量删除无确认（即使批量操作不需逐张确认，但显示清单）
• ❌ 导出时自动删除已导出照片（那是另一个功能，需要独立讨论）
￼
实现成本估算
层级            改动项                                                  工作量
数据库          新增 deleted_at 列 + migration                          小
仓库层          新增 soft_delete / restore / purge_expired 方法         小
API            POST /api/photos/trash + POST /api/photos/restore + DELETE /api/photos/{id}  中
后端逻辑        关联实体修复 + 缓存清理 + send2trash 集成                 中
前端            回收站 tab + 右键菜单 + 确认对话框                       中
测试            关联删除 + 恢复 + 边界条件                              中
总量估计：3–4 天。
￼
总评：删除是必要功能，但必须用回收站模式保护用户数据。建议作为 P2 优先级排在当前未完成的 P2 项之后。
